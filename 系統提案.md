## 1. 專案背景與動機

### 1.1 產業現況

網球教練與選手在賽後檢討時，往往仰賴人工反覆觀看比賽影片，手動記錄每拍擊球、回合長度、站位變化與落點分布。此流程不僅耗時費力，且受限於觀察者的主觀判斷，難以產出標準化、可複製的分析結果。

現有高階運動分析工具（如 Hawk-Eye、PlaySight）多依賴固定式多鏡頭攝影系統、嵌入式感測器或高價商用授權，導入成本高昂，使校隊、俱樂部與基層訓練單位難以採用。

### 1.2 機會識別

隨著深度學習在物件偵測（YOLO 系列）、姿態估計、多模態大語言模型（VLM）等領域的快速發展，以及 GPU 運算成本持續下降，現已具備以單一一般比賽影片作為輸入、自動產出具戰術價值之分析結果的技術條件。

### 1.3 提案核心概念

本系統旨在打造一套 **「上傳即分析」** 的網球影片分析平台——使用者僅需上傳一段普通比賽影片，系統即自動完成球體追蹤、球場偵測、球員識別、回合切割、事件分類、球速計算、落點分析與站位統計，並以互動式圖表呈現，同時提供基於比賽數據的 AI 問答服務。

## 2. 問題陳述

| 問題 | 影響 | 現有解決方案的不足 |
|---|---|---|
| 賽後分析依賴人工逐幀觀看 | 教練單場影片分析耗時 1–3 小時 | 無法規模化，難以跨場次比較 |
| 缺乏量化數據支撐訓練決策 | 「感覺型」教學，難以追蹤進步軌跡 | 手動記錄不一致、無標準化格式 |

| 觀看影片後缺乏即時摘要與解讀 | 關鍵資訊散落在腦中，難以快速回顧 | 無整合式問答與比賽數據連結機制 |

## 3. 系統目標

### 3.1 核心目標

1. **自動化分析**：使用者上傳影片後，系統自動完成從偵測到統計的完整分析流程，無需人工介入。
2. **量化呈現**：將影像分析結果轉換為具實際量測意義的數據（球速 km/h、落點公尺座標、站位區域），以互動式圖表呈現。
3. **智慧問答**：結合結構化賽事數據與 RAG 技術，提供以實際比賽內容為基礎的 AI 對話服務。
4. **低導入門檻**：不要求專用攝影設備或感測器，一般手機或攝影機錄製的比賽影片即可使用。

### 3.2 量化指標

| 指標 | 目標值 |
|---|---|
| 影片上傳至分析完成時間（1 分鐘影片） | ≤ 5 分鐘 |
| 球體偵測召回率 | ≥ 80% |
| 回合切割準確率 | ≥ 85% |
| 場地關鍵點偵測成功率 | ≥ 90%（正規場地、固定機位） |
| AI 問答回應延遲（首 token） | ≤ 3 秒 |
| 系統可同時服務使用者數 | ≥ 5（分析任務排隊） |

## 5. 功能性需求

### FR-1 影片管理

| 編號 | 需求描述 | 驗收條件 |
|---|---|---|
| FR-1.1 | 支援分塊上傳大型影片檔（≥ 500MB） | 10MB chunk、3 並發，上傳中斷可續傳 |
| FR-1.2 | 自動偵測非 H.264 編碼並背景轉碼 | AV1/VP9/VP8/Theora → H.264，轉碼期間顯示等待狀態 |
| FR-1.3 | 支援常見影片格式 | .mp4, .mov, .avi, .mkv, .webm |
| FR-1.4 | 登入用戶可管理歷史影片（列表、載入、刪除） | 軟刪除，檔案實體同步清除 |
| FR-1.5 | 訪客模式：無需註冊即可上傳與分析 | 7 天後自動清理影片與記錄 |

### FR-2 綜合分析

| 編號 | 需求描述 | 驗收條件 |
|---|---|---|
| FR-2.1 | 自動偵測球場 14 關鍵點並建立 Homography | 偵測失敗幀標記為過場，不中斷流程 |
| FR-2.2 | 自動追蹤球體位置（Kalman 濾波 + ROI 搜尋） | 靜止假陽性排除，滑動窗口插值 |
| FR-2.3 | 自動偵測雙方球員位置與姿態 | 區分上方（遠端）/ 下方（近端）球員 |
| FR-2.4 | 自動識別擊球事件與彈跳事件 | 速度方向反轉偵測 + VLM 輔助分類 |
| FR-2.5 | 自動切割比賽回合 | 基於驗證後的 contact 序列進行回合分段 |
| FR-2.6 | 自動計算球速（km/h） | 基於 Homography 世界座標，場地未偵測時填 null |
| FR-2.7 | 自動判定每拍球員站位區域 | net / service / baseline 三區分類 |
| FR-2.8 | VLM 輔助驗證擊球類型與回合勝負 | hit/bounce/unknown 分類；top/bottom/unknown 勝負判定 |
| FR-2.9 | 輸出標注影片（球框、骨架、場地線疊加） | 與原影片同目錄，H.264 MP4 格式 |
| FR-2.10 | 輸出結構化分析 JSON | 包含 metadata、summary、rallies、heatmap 四大區塊 |
| FR-2.11 | 即時進度回報 | 前端每 700ms 輪詢，顯示 0–100% 進度與階段描述 |

## 8. 技術選型與評估

| 需求面向 | 選用技術 | 選型理由 |
|---|---|---|
| **前端框架** | Next.js 16 + React 19 | Server-side rewrites 簡化反向代理；React 生態成熟；TypeScript 確保型別安全 |
| **後端框架** | FastAPI | 原生非同步支援 async/await；自動 OpenAPI 文件；Python 生態豐富（CV/ML） |
| **資料庫** | MySQL 8.0 | 穩定可靠；Docker 官方映像支援完善；索引效能滿足當前規模 |
| **ORM** | SQLAlchemy | Python 生態標準 ORM；支援原生 SQL 退路 |
| **物件偵測** | Ultralytics YOLO | 即時推論效能優異；社群活躍；支援 pose estimation 與自訂訓練 |
| **球體追蹤** | Kalman 濾波 + ROI 搜尋 | 低延遲即時追蹤；搭配靜止黑名單機制降低假陽性 |
| **座標轉換** | OpenCV Homography | 14 關鍵點 → 世界座標投影，成熟穩定 |
| **影片處理** | FFmpeg + decord | FFmpeg 處理編碼/轉碼；decord 高效逐幀解碼 |
| **VLM 推論** | vLLM + Qwen 多模態 | OpenAI 相容 API；GPU 記憶體效率高；支援多圖輸入 |
| **部署** | Docker Compose | 一鍵啟動；服務隔離；環境一致性 |
| **對外存取** | Cloudflare Tunnel | 無需公網 IP 或 SSL 憑證管理；零配置 HTTPS |

## 9. 資料流程設計

### 9.1 影片上傳流程

```
使用者選擇檔案
  → 前端分割為 10MB chunks
  → 3 並發 POST /api/upload_chunk
  → 全部完成 POST /api/upload_complete
  → 後端合併 chunks → 儲存至 videos/{owner|guest}/
  → 解析影片 metadata（duration, fps, codec, resolution）
  → 寫入 DB（analysis_records）
  → 若非 H.264 → 背景轉碼
  → 回傳 session_id + metadata
```

### 9.2 綜合分析流程

```
使用者點擊「綜合分析」
  → POST /api/analyze_combine
  → 後端 asyncio.create_task() 非同步執行：
      Phase 1（0–70%）：逐幀偵測
        ├─ 場地關鍵點 → Homography
        ├─ 球員姿態偵測
        ├─ 球體追蹤（Kalman + ROI）
        └─ 同步輸出標注影片
      Phase 2（70–72%）：後處理
        ├─ 事件偵測（contacts）
        ├─ 軌跡插值
        └─ 球速計算
      Phase 3（72–95%）：VLM 驗證
        ├─ 逐 contact 驗證（hit/bounce/unknown）
        ├─ 回合切割
        └─ 逐回合勝負判定
      Phase 4（95–100%）：統計彙整
        └─ 輸出 analysis JSON
  → 前端每 700ms 輪詢 GET /api/status/{session_id}
  → 完成後載入分析結果 + 標注影片
```

### 9.3 AI 聊天流程

```
使用者輸入問題
  → POST /api/chat {session_id, question}
  → 後端組建 messages：
      ├─ system：角色設定 + RAG 結構化摘要
      ├─ history：最近 10 輪對話
      └─ user：影片 URL + 問題文字（多模態）
  → POST vLLM /v1/chat/completions（stream=true）
  → 過濾 thinking tokens → SSE 串流回應
  → 前端逐 token 渲染
  → 完成後持久化至 DB
```

## 10. AI 模型策略

### 10.1 模型清單

| 模型 | 用途 | 來源 | 輸入 | 輸出 |
|---|---|---|---|---|
| YOLOv8（自訓練） | 網球偵測 | 自訓練 | 單幀影像 | 球體 bounding box |
| YOLO26n-pose（預訓練） | 球員姿態偵測 | COCO 17 關鍵點預訓練 | 單幀影像 | 人體骨架 + bounding box |
| YOLO26n-pose（fine-tune） | 場地關鍵點偵測 | 自訓練 14 關鍵點 | 單幀影像 | 場地 14 關鍵點座標 |
| Qwen 多模態模型 | VLM 驗證 + AI 問答 | 預訓練（Hugging Face） | 多張圖片 + 文字 | 結構化 XML / 自然語言 |

### 10.2 推論策略

- **YOLO 模型**：啟動時一次載入至 GPU，所有分析任務共享模型實例
- **VLM 模型**：透過 vLLM 以 OpenAI 相容 API 提供服務，支援 GPU 記憶體利用率調控（85%）
- **容錯設計**：VLM 呼叫失敗時，Phase 3 整體 try/except 保護，退回原始偵測結果

### 10.3 模型迭代方向

- 球體偵測：擴充訓練集（不同場地、光線、攝影角度），提升泛用性
- 場地偵測：增加室內場地與非標準角度訓練資料
- 擊球類型：從 4 類（serve/overhead/swing/unknown）擴充至更細緻分類
- 雙打支援：球員偵測模型需擴充至 4 人場景

## 11. 安全性設計

### 11.1 認證與授權

| 機制 | 說明 |
|---|---|
| JWT（HS256） | 24 小時有效期，payload 僅含 user_id |
| bcrypt | 密碼雜湊儲存，不可逆向還原 |
| 訪客 token | 64-char hex，sessionStorage 儲存，關閉分頁即失效 |
| session 存取控制 | owner_id 比對（登入用戶）或 guest_token 比對（訪客） |

### 11.2 資料安全

| 威脅 | 對策 |
|---|---|
| Path Traversal | `assert_under_video_dir()` 確保所有檔案操作在 VIDEO_DIR 內 |
| 未授權存取他人影片 | session owner 驗證 + DB owner_id 欄位比對 |
| JWT 密鑰洩漏 | SECRET_KEY 由環境變數注入，未設定時啟動警告 |
| 檔案殘留 | 軟刪除同步清除實體檔案；訪客資料 7 天自動清理 |

### 11.3 網路安全

- Cloudflare Tunnel 提供 HTTPS 加密傳輸
- 後端服務僅暴露於 Docker 內部網路，不直接對外
- CORS 設定（目前全開，正式環境應限縮）

## 12. 部署與維運策略

### 12.1 部署方式

- **容器化**：全服務 Docker Compose 編排
- **環境切換**：`DEV_MODE` 環境變數控制開發/生產模式
- **VLM 可選**：`--profile vllm` 選擇性啟動，降低無 GPU 環境的部署門檻
- **資料持久化**：MySQL 與 Hugging Face 模型快取均掛載 host volume

### 12.2 硬體建議

| 等級 | CPU | RAM | GPU | 儲存 | 適用場景 |
|---|---|---|---|---|---|
| 基礎 | 8 核心 | 16GB | 12GB VRAM | 100GB SSD | 僅 YOLO 分析，無 VLM |
| 建議 | 8+ 核心 | 32GB | 24GB+ VRAM | 200GB SSD | 完整功能含 VLM 與 RAG 問答 |

### 12.3 自動維運

| 任務 | 頻率 | 說明 |
|---|---|---|
| chunk 暫存清理 | 每 10 分鐘 | 刪除超過 1 小時的上傳分塊 |
| 訪客資料清理 | 每 10 分鐘 | 刪除超過 7 天的訪客影片與 DB 記錄 |
| 健康檢查 | 持續 | MySQL healthcheck 確保 DB 就緒 |

## 14. 風險評估與對策

| 風險 | 可能性 | 影響度 | 對策 |
|---|---|---|---|
| 非標準攝影角度導致場地偵測失敗 | 高 | 中 | 失敗幀自動標記為過場，不中斷分析；逐步擴充訓練資料 |
| GPU 記憶體不足（同時運行 YOLO + VLM） | 中 | 高 | VLM 以獨立 profile 部署，可選擇性啟用；分析與 VLM 分階段使用 GPU |
| 高速球體偵測遺漏 | 高 | 中 | Kalman 預測 + ROI 搜尋 + 滑動窗口插值補償；持續擴充訓練集 |
| VLM 回應不穩定或超時 | 中 | 低 | 60 秒 timeout + Phase 3 整體 try/except 容錯 |
| 大量訪客同時使用導致資源不足 | 低 | 高 | 分析任務排隊機制；訪客資料定期清理；未來可加入佇列限流 |
| SECRET_KEY 未設定導致 JWT 跨重啟失效 | 中 | 低 | 啟動時自動警告；文件明確提示需設定環境變數 |
